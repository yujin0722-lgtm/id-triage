## 2. プロジェクトの目的

感染症に関する臨床疑問から、PubMed検索式を組み立て、検索結果を研究デザイン別に整理するウェブアプリを作成する。

単純に論文を検索するだけではなく、以下の作業を支援する。

1. 感染症領域で使われる検索語や言い換え表現を検索式に追加する
2. 作成されたPubMed検索式を利用者が確認・修正できるようにする
3. 検索結果を研究デザイン別に分類する
4. 各論文を「読む」「保留」「除外」に振り分ける
5. 除外理由を簡単に記録する

本アプリは、PubMedの代替ではなく、**PubMed検索を始める前後の整理を支援する補助ツール**と位置づける。

---

## 3. 解決したい問題

PubMedで感染症関連の論文を探す際には、次のような問題がある。

* 適切な英語表現や類義語がすぐに思いつかない
* 「治療期間」「経口スイッチ」などの概念に複数の表現がある
* 検索結果にガイドライン、RCT、観察研究、症例報告が混在する
* 検索結果を上から順に確認するだけでは効率が悪い
* 一度確認した論文や除外した論文を再び開いてしまう
* なぜその論文を除外したのか記録が残らない

本アプリでは、検索語の展開、研究デザイン別分類、スクリーニング状況の保存によって、これらの負担を軽減する。

## 4. 想定利用者

主な対象者は以下とする。

* 感染症診療に関わる医師
* 臨床疑問について簡単な文献検索を行いたい医療従事者
* PubMed検索式の作成に慣れていない人
* EBMの学習者
* 勉強会や抄読会のために論文を探す人

初期版では日本語の画面を使用するが、PubMedに送信する主題の入力は原則として英語とする。

## 5. アプリの中心的な価値

本アプリの中心的な価値を、次の一文で定義する。

> 感染症に特有の臨床テーマからPubMed検索式を作り、検索結果を研究デザイン別に整理して、読むべき論文の選別を支援する。

本アプリは、論文内容の正しさや推奨の強さを自動判定するものではない。

## 7. システム構成

### 7.1 フロントエンド

以下の技術を使用する。

* HTML
* CSS
* JavaScript
* JavaScript Modules
* GitHub Pages

React、Vue、Next.jsなどのフレームワークは初期版では使用しない。

GitHub Pagesは、GitHubリポジトリ内のHTML、CSS、JavaScriptを静的ウェブサイトとして公開できるため、本アプリのフロントエンド公開先として使用する。

### 7.2 外部API

NCBI E-utilitiesを使用する。

基本的な通信は次の2段階とする。

1. `ESearch`で検索条件に一致するPMIDを取得する
2. `ESummary`で各PMIDのタイトル、著者、雑誌名、発表日などを取得する

ESearchは検索条件に一致するIDの取得、ESummaryはIDに対応する文献概要の取得、EFetchは抄録を含む詳細情報の取得に使用できる。

初期版では、原則としてESearchとESummaryのみを使用する。抄録取得が必要になった段階でEFetchを追加する。

### 7.3 API接続方式

初期プロトタイプでは、GitHub Pages上のJavaScriptからNCBI E-utilitiesを直接呼び出す。

API通信部分は専用ファイルに分離し、将来的に以下の構成へ変更できるようにする。

```text
GitHub Pages
    ↓
Cloudflare Workersなどの中継API
    ↓
NCBI E-utilities
```

中継APIへの移行を想定する理由は以下である。

* APIキーをブラウザ側に公開しないため
* NCBIへのアクセス頻度を管理するため
* キャッシュを導入するため
* API障害時の処理を統一するため

ただし、中継APIは初期版の必須要件には含めない。

## 9. 検索条件入力画面

### 9.1 病原体・疾患・病態

単純な疾患名、病原体名、病態名を入力する自由入力欄を1つ設ける。

表示名：

```text
病原体・疾患・病態
```

入力例：

```text
Staphylococcus aureus bacteremia
infective endocarditis
Clostridioides difficile infection
community-acquired pneumonia
```

初期版では英語入力を基本とする。利用者がPubMedの検索構文を直接記述する欄ではなく、通常の疾患名や病態名を入力する欄とする。

入力後、アプリは主題検索語辞書を参照し、登録された類義語、言い換え、略語を含む主題検索ブロックを作成する。

辞書に一致する項目がない場合は、入力語を勝手に推測して別の医学用語へ置き換えず、入力内容をPubMedの検索語として使用する。その場合は「辞書に一致する検索語がないため、入力語をそのまま使用した」と表示する。

未入力の場合は検索式の作成および検索を実行できないようにする。

#### 変換内容の表示

検索式の作成時に、次の情報を検索式とは別の領域へ表示する。

```text
入力した語：Staphylococcus aureus bacteremia
辞書との一致：あり
追加した表現：
・Staphylococcus aureus bacteremia
・S. aureus bacteremia
・Staphylococcus aureus bloodstream infection

作成した主題検索ブロック：
(
  "Staphylococcus aureus bacteremia"[Title/Abstract]
  OR "S. aureus bacteremia"[Title/Abstract]
  OR "Staphylococcus aureus bloodstream infection"[Title/Abstract]
)
```

変換内容は、アプリが追加した語と利用者が入力した語を区別できる表示とする。

### 9.2 臨床テーマ

以下から1つ選択する。

| 表示名       | 内部値                      |
| --------- | ------------------------ |
| 指定なし      | none                     |
| 治療全般      | treatment                |
| 治療期間      | treatment-duration       |
| 経口薬への切り替え | oral-switch              |
| ソースコントロール | source-control           |
| 血液培養陰性化   | culture-clearance        |
| 診断        | diagnosis                |
| 迅速診断      | rapid-diagnosis          |
| 予後        | prognosis                |
| 再発        | recurrence               |
| 薬剤耐性      | antimicrobial-resistance |
| 予防        | prevention               |

初期版では単一選択とする。

複数テーマの同時選択は将来版で検討する。

### 9.3 研究デザイン

複数選択可能なチェックボックスとする。

* ガイドライン
* システマティックレビュー
* メタ解析
* ランダム化比較試験
* 観察研究
* 症例報告

チェックボックスはすべて未選択にできる。何も選択しなかった場合は、研究デザインによる検索条件を追加しない。

「指定なし」のチェックボックスは設けない。

### 9.4 発表年

以下を入力できるようにする。

* 開始年
* 終了年

初期値は次のとおりとする。

```text
開始年：空欄
終了年：現在の年
```

開始年が終了年より後の場合は、検索を実行せずエラーを表示する。

### 9.5 表示件数

以下から選択する。

* 10件
* 20件
* 50件

初期値は20件とする。

### 9.6 並び順

以下から選択する。

* 関連度順
* 新しい順

初期値は関連度順とする。

## 10. 検索語辞書

検索語辞書は、主題検索語辞書と臨床テーマ検索語辞書に分けて管理する。

検索語は画面のHTMLに直接記載せず、独立した`search-terms.js`で管理する。

### 10.1 主題検索語辞書

単純な疾患名、病原体名、病態名から、PubMedへ送る主題検索ブロックを作成するために使用する。

初期版では辞書に登録された語だけを展開する。自由入力から医学的な類義語を自動推測する処理は行わない。

照合時には、大文字・小文字、前後の空白、連続する空白の違いを正規化する。正規化後に完全一致した辞書項目を使用する。

例：

```javascript
const subjectTerms = {
  "staphylococcus aureus bacteremia": {
    label: "Staphylococcus aureus bacteremia",
    terms: [
      '"Staphylococcus aureus bacteremia"[Title/Abstract]',
      '"S. aureus bacteremia"[Title/Abstract]',
      '"Staphylococcus aureus bloodstream infection"[Title/Abstract]'
    ]
  }
};
```

辞書に一致しない場合は、入力語を括弧で囲んだ検索ブロックとして使用する。

```text
入力：unregistered disease name
変換：(unregistered disease name)
説明：辞書に一致する検索語がないため、入力語をそのまま使用しました。
```

### 10.2 臨床テーマ検索語辞書

臨床テーマには、あらかじめ複数の検索語を登録する。

例：

```javascript
const clinicalThemes = {
  treatmentDuration: [
    '"treatment duration"[Title/Abstract]',
    '"duration of therapy"[Title/Abstract]',
    '"antibiotic duration"[Title/Abstract]',
    '"duration of antimicrobial therapy"[Title/Abstract]'
  ],

  oralSwitch: [
    '"oral switch"[Title/Abstract]',
    '"oral step-down"[Title/Abstract]',
    '"intravenous-to-oral"[Title/Abstract]',
    '"IV-to-oral"[Title/Abstract]'
  ],

  sourceControl: [
    '"source control"[Title/Abstract]',
    '"infectious source control"[Title/Abstract]'
  ],

  cultureClearance: [
    '"culture clearance"[Title/Abstract]',
    '"blood culture clearance"[Title/Abstract]',
    '"clearance of bacteremia"[Title/Abstract]',
    '"time to blood culture clearance"[Title/Abstract]'
  ]
};
```

これらの語は、初期実装後に実際の検索結果を確認して調整する。

辞書の変更だけで検索式を改善できる構造とし、画面表示やAPI処理のコードを書き換えなくてもよいようにする。

## 11. PubMed検索式生成

### 11.1 基本構造

検索式は、次の順に組み立てる。

```text
主題
AND 臨床テーマ
AND 研究デザイン
AND 発表年
```

例：

入力条件：

```text
主題：
Staphylococcus aureus bacteremia

臨床テーマ：
治療期間

研究デザイン：
システマティックレビュー、メタ解析

発表年：
2020～2026
```

生成例：

```text
(
  "Staphylococcus aureus bacteremia"[Title/Abstract]
  OR "S. aureus bacteremia"[Title/Abstract]
  OR "Staphylococcus aureus bloodstream infection"[Title/Abstract]
)
AND
(
  "treatment duration"[Title/Abstract]
  OR "duration of therapy"[Title/Abstract]
  OR "antibiotic duration"[Title/Abstract]
  OR "duration of antimicrobial therapy"[Title/Abstract]
)
AND
(
  systematic review[Publication Type]
  OR meta-analysis[Publication Type]
)
AND
2020:2026[Publication Date]
```

### 11.2 検索語の変換内容

生成された検索式の上部に、検索語の変換内容を表示する。

表示項目：

* 利用者が入力した主題
* 辞書に一致したか
* アプリが追加した類義語・言い換え・略語
* 作成された主題検索ブロック
* 選択した臨床テーマから追加された検索語
* 選択した研究デザインから追加された検索条件
* 発表年から追加された検索条件

各項目は「利用者の入力」と「アプリによる追加」を明確に区別する。

### 11.3 検索式の確認

生成した検索式は、検索実行前にテキストエリアへ表示する。

利用者は検索式を直接修正できる。

以下のボタンを設置する。

* 検索式を作成
* PubMedを検索
* 入力内容に戻す
* 検索式をコピー
* 検索条件をクリア

「検索式を作成」と「PubMedを検索」は分ける。

これにより、利用者は実際にAPIを呼び出す前に、どのような検索式が作成されたか確認できる。

### 11.4 手動修正後の扱い

利用者が生成された検索式を手動で変更した場合、APIには修正後の検索式を送信する。

手動修正が行われた場合は、テキストエリア付近に次の表示を行う。

```text
検索式は手動で修正されています。
```

再度「検索式を作成」を押した場合は、現在の入力条件から検索式を作り直し、手動修正後の内容を上書きする。

### 11.5 前回内容の復元

ページを再読み込みした場合は、保存されている次の内容を復元する。

* 主題の入力内容
* 臨床テーマ
* 研究デザイン
* 開始年と終了年
* 表示件数
* 並び順
* 主題検索語の変換内容
* 自動生成時の検索式
* 手動修正後の検索式
* 手動修正の有無

手動修正後の検索式が保存されている場合は、その検索式をテキストエリアへ復元する。

復元時は次のメッセージを表示する。

```text
前回の検索条件と検索式を復元しました。
```

「検索条件をクリア」を押した場合は、現在の入力欄、生成された検索式、手動修正後の検索式、および`last-search`の保存データを削除する。論文ごとのスクリーニング判定は削除しない。

## 12. PubMed API通信

### 12.1 ESearch

使用目的：

* 検索結果の総件数取得
* 表示対象となるPMIDの取得

主なパラメータ：

```text
db=pubmed
term=検索式
retmode=json
retmax=表示件数
sort=relevance または pub_date
```

### 12.2 ESummary

使用目的：

* PMID
* 論文タイトル
* 著者
* 雑誌名
* 発表日
* 巻・号・ページ
* DOI
* Publication Type
* PubMedへのリンク

ESearchで得られたPMIDをまとめてESummaryに送信する。

1論文ごとに個別のAPIリクエストは行わない。

#### 書誌情報の欠損への対応

論文カードを通常データとして扱う最低条件は、PMIDと論文タイトルを取得できることである。

著者、雑誌名、発表日、巻、号、ページ、DOI、Publication Typeは任意項目として扱い、欠けていても他の情報を表示する。

* DOIがない場合はDOI欄を表示しない
* 著者がない場合は「著者情報なし」と表示する
* 雑誌名や発表年がない場合は取得できた項目だけを表示する
* Publication Typeがない場合は研究デザインを「その他」に分類する
* PMIDまたはタイトルが取得できない場合は、通常カードとは分けて「書誌情報を十分に取得できませんでした」と表示し、他の論文の表示処理は継続する

抄録またはより詳細なPublication Type情報が必要になった場合は、将来版でEFetchの追加を検討する。

### 12.3 APIリクエスト数

1回の検索につき、原則として次の2リクエストとする。

```text
1. ESearch
2. ESummary
```

NCBIは、APIキーを使用しない場合は毎秒3リクエストを超えないことを求めている。APIキーを使用した場合の標準上限は毎秒10リクエストである。初期版ではAPIキーを使用せず、検索ボタンの連打防止とリクエストの直列実行によって制限を守る。

### 12.4 二重実行の防止

検索開始後は、通信が終了するまで検索ボタンを無効化する。

通信中は以下を表示する。

```text
PubMedを検索しています……
```

成功、失敗、タイムアウトのいずれかでボタンを再度有効にする。

* 著者

* 発表年
* PMID
* DOI
* PubMedで開くボタン
* 読むボタン
* 保留ボタン
* 除外ボタン

表示例：

```text

PMID: 12345678
        
        
DOI: 10.xxxx/xxxx
```

## 14. 研究デザイン別分類

### 14.1 分類カテゴリー

論文を以下のカテゴリーに分類する。

1. ガイドライン・統合研究
2. 介入研究
3. 観察研究
4. 症例報告
5. その他

### 14.2 分類の優先順位

1つの論文に複数のPublication Typeが付いている場合は、次の優先順位で1つのカテゴリーへ分類する。

```text
ガイドライン
↓
システマティックレビュー・メタ解析
↓
ランダム化比較試験・臨床試験
↓
観察研究
↓
症例報告
↓
その他
```

分類はPubMedから取得したPublication Typeだけを根拠として行う。タイトルや雑誌名から研究デザインを推測しない。

Publication Typeに対応する明確な分類がない場合、またはPublication Typeを取得できない場合は「その他」とする。

この分類は論文の質を判定するものではなく、検索結果を整理するための便宜的分類である。

### 14.3 表示方式

カテゴリーごとに見出しを作る。

```text
ガイドライン・統合研究　4件
介入研究　　　　　　　　3件
観察研究　　　　　　　　9件
症例報告　　　　　　　　2件
その他　　　　　　　　　2件
```

各カテゴリーは開閉できるようにする。

初期表示ではすべて開いた状態とする。

## 15. 簡易スクリーニング機能

### 15.1 判定

各論文について以下から1つを選択する。

* 未判定
* 読む
* 保留
* 除外

同じ論文で複数の状態を同時に選択することはできない。

### 15.2 除外理由

「除外」を選択した場合は、以下の選択肢を表示する。

* 対象患者が違う
* 病原体・疾患が違う
* 介入が違う
* 評価項目が違う
* 研究デザインが目的と違う
* 症例報告
* 古すぎる
* 重複
* その他

「その他」を選択した場合は、自由記載欄を表示する。

### 15.3 判定済み論文の表示

カードの左側または上部に判定状態を表示する。

```text
［読む］
［保留］
［除外：対象患者が違う］
```

検索条件を変えて同じPMIDが再度表示された場合も、過去の判定を復元する。

* 開始年と終了年

* 並び順

* 自動生成された検索式

保存キーにはバージョン番号を付ける。

* 前回の検索条件・検索式が保存されているか

* 論文判定だけをすべて削除する操作
* 検索条件、手動修正後の検索式、論文判定をまとめて削除する操作

削除前には確認メッセージを表示し、削除後に元へ戻せないことを明示する。

## 22. NCBI利用方針への対応

NCBIの一般的な利用要件に対応するため、次の方針を採用する。

* E-utilities専用URLを使用する
* APIリクエストを毎秒3回以下にする
* 1論文ごとの個別リクエストを避ける
* 複数PMIDをまとめてESummaryへ送る
* 検索ボタンの連打を防止する
* アプリ名を`tool`パラメータとして送信する
* 公開運用時にはプロジェクト用連絡先を`email`パラメータとして設定する
* APIキーを使用する場合は中継API側で管理する

NCBIの公式資料では、E-utilitiesの各リクエストに`tool`と`email`を含めることが推奨されている。

プロジェクト用の連絡先が決まるまでは、`email`パラメータをコードに仮設定しない。
